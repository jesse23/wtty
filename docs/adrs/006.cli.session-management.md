# ADR 006: CLI — Session Management Commands

**SPEC:** [cli](../specs/cli.md)
**Status:** Accepted
**Date:** 2026-03-22

---

## Context

ADR 004 defines the REST API for session CRUD and rename. The CLI needs to expose that API as four user-facing commands: `webtty run`, `webtty ls`, `webtty rm`, and `webtty rename`. Each command is a thin HTTP wrapper, but several design questions need decisions:

1. **`webtty run`** — should it auto-start the server if not running, and reuse an existing session by ID?
2. **`webtty ls`** — what does the output look like? How is "server not running" surfaced?
3. **`webtty rm`** — any confirmation prompt before killing a session?
4. **`webtty rename`** — positional args or flags?
5. **Commander.js** — ADR 002 deferred it; is it justified now?

## Decision

**`webtty run [id]`**: The primary entry point — works regardless of server state.

1. **Server check**: `GET /api/sessions`. If connection refused, start the server first (same logic as `webtty start` — spawn detached, poll until ready).
2. **Session**: if `id` is provided and already exists (`GET /api/sessions/:id` returns 200), skip creation. Otherwise `POST /api/sessions` with `{ id }` (or omit for auto-generated ID).
3. **Browser**: open `http://127.0.0.1:<PORT>/s/:id` in the default browser via `open` (macOS/Linux) / `start` (Windows).

This means `webtty run` is a safe one-shot command — running it cold (no server, no session) always ends with a browser tab open on a working terminal.

**`webtty ls`**: `GET /api/sessions`. Prints a table of all sessions (id, connected, createdAt). If connection refused, prints `webtty is not running` and exits 1. Empty list prints `no sessions` — not an error.

**`webtty rm <id>`**: `DELETE /api/sessions/:id`. No confirmation prompt — this is a CLI tool for developers, not a destructive operation on shared data. Prints `removed <id>` on success, `session <id> not found` on 404.

**`webtty rename <id> <new-id>`**: `PATCH /api/sessions/:id`. Two positional args — old ID then new ID. Prints `renamed <id> → <new-id>` on success. Errors: `session <id> not found` (404), `<new-id> already exists` (409), `invalid id: <new-id>` (400).

**Commander.js**: Introduced in this slice. Four subcommands with positional arguments justify the dependency. `process.argv` parsing does not scale beyond two commands cleanly.

## Considered Options

### Option A: Auto-start server + reuse existing session + open browser (chosen)

- **Pros**: True one-shot UX — `webtty run` always works, no prerequisite `webtty start`. Reusing an existing session by ID is the natural expectation (matches `docker run` reattach behaviour). Opening the browser removes the copy-paste step.
- **Cons**: `webtty run` does more than one thing — server start + session create/reuse + browser open. Mitigated by the fact that each step is idempotent and the happy path is transparent to the user.

### Option B: Require server to be running

- **Pros**: Simpler implementation — one HTTP call.
- **Cons**: Forces users to remember `webtty start` before `webtty run`. Defeats the purpose of a one-shot command.

### Option C: Confirmation prompt on `webtty rm`

- **Pros**: Prevents accidental deletion.
- **Cons**: Breaks pipe/script usage. Sessions are memory-only and easily recreated. `webtty run` opens a new one in one command.

### Option D: `--name` flag instead of positional ID for `webtty run`

`webtty run --name my-session` instead of `webtty run my-session`.

- **Pros**: More explicit, matches Docker/AWS flag style.
- **Cons**: More typing for the common case. Positional is idiomatic for kubectl and tmux (`tmux new-session -s name`). Consistent with `webtty rm <id>` and `webtty rename <id> <new-id>`.

## Consequences

- `webtty run` is a true one-shot command — works cold (no server, no session) and always ends with a browser tab on a working terminal
- If the server was already running and the session already exists, `webtty run <id>` just opens a new browser tab — no side effects
- `webtty ls` doubles as a health check — connection refused surfaces the server status without a separate command
- No confirmation on `rm` — consistent with `rm` convention; sessions are cheap to recreate
- Commander.js added as a runtime dependency in this slice
- `--no-open` flag for `webtty run` deferred to a future slice
