# ADR 006: CLI — Session Management Commands

**SPEC:** [cli](../specs/cli.md)
**Status:** Accepted
**Date:** 2026-03-22

---

## Context

ADR 004 defines the REST API for session CRUD and rename. The CLI needs to expose that API as four user-facing commands: `webtty run`, `webtty ls`, `webtty rm`, and `webtty rename`. Each command is a thin wrapper — one HTTP call — but several design questions need decisions:

1. **`webtty run`** — should it open the browser automatically, or just print the URL?
2. **`webtty ls`** — what does the output look like? How is "server not running" surfaced?
3. **`webtty rm`** — any confirmation prompt before killing a session?
4. **`webtty rename`** — positional args or flags?
5. **Commander.js** — ADR 002 deferred it; is it justified now?

## Decision

**`webtty run [id]`**: `POST /api/sessions`. Auto-generates ID if omitted. On success, prints the session URL (`http://127.0.0.1:<PORT>/s/:id`) and opens it in the default browser via `open` (macOS/Linux) / `start` (Windows) — same cross-platform approach used by Vite. If the server is not running, prints `webtty is not running`.

**`webtty ls`**: `GET /api/sessions`. Prints a table of all sessions (id, connected, createdAt). If connection refused, prints `webtty is not running` and exits 1. Empty list prints `no sessions` — not an error.

**`webtty rm <id>`**: `DELETE /api/sessions/:id`. No confirmation prompt — this is a CLI tool for developers, not a destructive operation on shared data. Prints `removed <id>` on success, `session <id> not found` on 404.

**`webtty rename <id> <new-id>`**: `PATCH /api/sessions/:id`. Two positional args — old ID then new ID. Prints `renamed <id> → <new-id>` on success. Errors: `session <id> not found` (404), `<new-id> already exists` (409), `invalid id: <new-id>` (400).

**Commander.js**: Introduced in this slice. Four subcommands with positional arguments justify the dependency. `process.argv` parsing does not scale beyond two commands cleanly.

## Considered Options

### Option A: Open browser automatically on `webtty run` (chosen)

- **Pros**: Single command to create and open — matches `docker run` UX. Consistent with the spec intent ("opens the session URL in the default browser").
- **Cons**: Undesirable in CI/headless environments — mitigated by `--no-open` flag (deferred to a future slice).

### Option B: Print URL only, no browser open

- **Pros**: Simpler, no platform detection needed.
- **Cons**: Loses the key UX benefit of `webtty run` over `curl`. User has to copy-paste the URL.

### Option C: Confirmation prompt on `webtty rm`

- **Pros**: Prevents accidental deletion.
- **Cons**: Breaks pipe/script usage. Sessions are memory-only and easily recreated. `webtty run` opens a new one in one command.

### Option D: `--name` flag instead of positional ID for `webtty run`

`webtty run --name my-session` instead of `webtty run my-session`.

- **Pros**: More explicit, matches Docker/AWS flag style.
- **Cons**: More typing for the common case. Positional is idiomatic for kubectl and tmux (`tmux new-session -s name`). Consistent with `webtty rm <id>` and `webtty rename <id> <new-id>`.

## Consequences

- `webtty run` is the primary entry point for new users — one command creates a session and opens it
- `webtty ls` doubles as a health check — connection refused surfaces the server status without a separate command
- No confirmation on `rm` — consistent with `rm` convention; sessions are cheap to recreate
- Commander.js added as a runtime dependency in this slice
- `--no-open` flag for `webtty run` deferred to a future slice
