# ADR 003: Release Process — semantic-release on GitHub

**SPEC:** [release-process](../specs/release-process.md)
**Status:** Proposed
**Date:** 2026-03-21

---

## Context

wtty has a stub release workflow (`.github/workflows/release.yml`) that runs `bun run build` then `echo "TODO - publish step"`. No versioning, no published artifacts.

wtty is distributed as a standalone binary — not published to npm (`"private": true`). A release means a GitHub Release with platform-specific binary artifacts attached. The version should be derived automatically from what changed, not set manually.

The cfx repo solved the same versioning problem with semantic-release (ADR 004): conventional commit prefixes in PR/MR titles drive version bumps post-merge, git tags are the sole version record, no release commits. That pattern maps cleanly to wtty with one difference: instead of `@semantic-release/npm`, the publish step is `@semantic-release/github` with binary assets.

## Decision

Use **semantic-release** with `@semantic-release/github`, running post-merge on `main`. Developers write a conventional commit prefix in the PR title. GitHub squash merge uses the PR title as the commit message on `main`. semantic-release reads that message, determines the version bump, creates a GitHub Release with release notes, and attaches pre-built binaries. No manual versioning, no release commits, no extra files per PR.

## Considered Options

### Option A: semantic-release post-merge on main, tag-only (chosen)

- **Pros**: Zero developer overhead beyond PR title. GitHub Release + binary upload fully automated. Battle-tested. No version conflicts between concurrent PRs. No release commits — clean one-commit-per-PR history on `main`. No branch push permissions needed (only tag push). `GITHUB_TOKEN` is provided automatically — no secrets to configure.
- **Cons**: `package.json` version doesn't reflect the released version (use `0.0.0-development` placeholder). No in-repo `CHANGELOG.md` (release notes live on GitHub Releases). Requires conventional commit discipline in PR titles (mitigate: squash merge uses PR title — one line to get right).

### Option B: Manual GitHub Release

Create GitHub Releases manually — edit the release notes, attach binaries by hand, tag manually.

- **Pros**: Full control over release notes. No tooling to configure.
- **Cons**: Manual — error-prone, inconsistent, easy to forget. Version tracking relies on human discipline. Doesn't scale.

### Option C: release-it

release-it (used in cfx ADR 006) supports GitHub releases natively and has a simpler config surface than semantic-release.

- **Pros**: Simpler per-package config, base version frozen on prerelease.
- **Cons**: wtty has no prerelease channel need (the main reason cfx switched to release-it). semantic-release's automatic version derivation from commit type is the better fit here — release-it requires either a manual bump type or more config to replicate the same behavior.

### Option D: Changesets

- **Pros**: Human-written changelogs, 1 dependency.
- **Cons**: Extra file per PR, developer must run `npx changeset`, changeset summary duplicates PR title. Same reasons rejected in cfx ADR 004.

## Consequences

**Good**: Zero developer overhead — write a conventional PR title, everything else is automated. GitHub Release auto-created with release notes. Binary artifacts attached. Clean `main` history — one commit per PR, no interleaved release commits. `GITHUB_TOKEN` is automatic — no manual secret setup. Follows semantic-release's own recommended approach (tag-only, no commit-back).

**Bad**: `package.json` version stays at `0.0.0-development` — doesn't reflect the released version. No `CHANGELOG.md` in repo (release notes on GitHub Releases). Requires conventional commit discipline in PR titles.

## Implementation Notes

### Plugin Chain

```json
{
  "branches": ["main"],
  "tagFormat": "v${version}",
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    ["@semantic-release/github", { "assets": ["dist/wtty-*"] }]
  ]
}
```

`@semantic-release/commit-analyzer`, `@semantic-release/release-notes-generator`, and `@semantic-release/github` are all bundled with `semantic-release` — no additional plugin packages needed. `@semantic-release/git` and `@semantic-release/changelog` are intentionally omitted (tag-only).

### Release Workflow

```yaml
# .github/workflows/release.yml
on:
  push:
    branches: [main]
jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
          persist-credentials: false
      - uses: oven-sh/setup-bun@v2
        with:
          bun-version: 1.3.10
      - run: bun install --frozen-lockfile
      - run: bun run build
      - name: Build platform binaries
        run: bun run build:binaries
      - name: Release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: bunx semantic-release
```

- `fetch-depth: 0` — full clone so semantic-release can walk commit history to find previous tags
- `persist-credentials: false` — semantic-release manages its own git auth via `GITHUB_TOKEN`
- Binary build precedes the release step so assets exist before `@semantic-release/github` uploads them

### Why Tag-Only (No Release Commits)

semantic-release's own FAQ recommends against committing during release. Omitting `@semantic-release/git` and `@semantic-release/changelog`:

- **No branch push permissions needed** — `GITHUB_TOKEN` only needs to push tags and create releases
- **Clean history** — one commit per PR, no interleaved `chore(release):` commits
- **No `[skip ci]` dance** — no recursive pipeline triggering
- **Fewer dependencies** — 1 devDependency (`semantic-release`) instead of 3+

### GitHub Repository Settings

| Setting | Value | Why |
|---------|-------|-----|
| Squash merging | Enabled | PR title becomes the commit semantic-release reads |
| Workflow permissions | Read and write | `GITHUB_TOKEN` needs to push tags and create releases |

### `package.json` Version Placeholder

```json
{ "version": "0.0.0-development" }
```

Signals that the in-repo version is not the released version. semantic-release writes the real version to `package.json` in CI only (during the npm plugin step) — but since there's no `@semantic-release/npm` here, it's purely cosmetic. Git tags are the source of truth.
