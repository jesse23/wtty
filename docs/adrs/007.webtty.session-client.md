# ADR 007: webtty — Session Client

**SPEC:** [webtty](../specs/webtty.md)
**Status:** Accepted
**Date:** 2026-03-22

---

## Context

When a browser tab is reloaded, the ghostty-web `Terminal` instance is destroyed and recreated from scratch — its scrollback buffer is gone. The PTY on the server remains alive (sessions survive WebSocket disconnects per ADR 005), so the reconnect succeeds, but the user sees a blank terminal with no history.

A second scenario: the user copies the session URL and opens it in a new tab. The original implementation evicts the existing WebSocket connection, causing a visible disconnect/reconnect flash in the first tab.

Four behaviours need to be defined:

1. **Scrollback replay** — how does the fresh terminal recover prior output?
2. **First-connect vs reconnect** — should the welcome banner appear on every connect?
3. **Shell exit** — should the session persist after the user types `exit`?
4. **Multi-tab** — can multiple tabs share the same session simultaneously?

## Decision

**Scrollback replay (server-side ring buffer)**

The server maintains a `scrollback: string` field per session — a ring buffer of raw PTY output capped at `SCROLLBACK_MAX` bytes (256 KB, ~3000 lines of 80-col output). Every byte from `pty.onData` is appended and the buffer is trimmed with `.slice(-SCROLLBACK_MAX)`.

On reconnect (PTY already exists), the server sends the full scrollback buffer to the fresh WebSocket before any new output arrives. Because the buffer contains raw terminal escape sequences (including alternate-screen sequences like `\x1b[?1049h`), the terminal reconstructs its state correctly — full-screen TUI apps (vim, htop, Claude Code) repaint as expected.

**First-connect vs reconnect**

The welcome banner is shown only on first connect (when `session.pty` is `null`). The banner is also written into `session.scrollback` directly so it appears on replay. On reconnect, no banner is sent — the scrollback replay alone is sufficient.

A `resize` (SIGWINCH) is sent to the PTY immediately after the scrollback replay on every reconnect. This causes TUI apps to redraw their full-screen state on top of the replayed content, ensuring a clean screen for apps like vim that use alternate screen mode. The client also sends an `onopen` resize JSON message, so SIGWINCH fires twice in quick succession — harmless.

**Shell exit closes the session and all tabs**

When the PTY exits (e.g. user types `exit`), the server:
1. Deletes the session from the registry immediately
2. Closes all connected WebSockets with code `4001`

The client's existing `4001` handler (introduced in ADR 005 for deleted sessions) writes "Session removed." and calls `window.close()` after 500ms. No delay is added before closing — the user explicitly typed `exit`, so immediate closure is the right behaviour. All open tabs for that session close together.

Server stop (`webtty stop` or SIGINT) sends close code `1001` instead. The client writes "Server stopped." and also calls `window.close()` after 500ms.

**`window.close()` browser restriction**: browsers only permit `window.close()` on tabs that were opened via `window.open()` or duplicated from such a tab. Tabs opened by the OS `open`/`xdg-open` command or by the user typing a URL directly are treated as unowned — `window.close()` is silently ignored with a console warning. In practice this means:
- Tabs opened by `webtty run` and tabs duplicated from them → close automatically ✅
- Tabs opened by manually navigating to `localhost:2346` → show "Server stopped." / "Session removed." but remain open ⚠️

There is no JS workaround for this restriction. It is browser-enforced by design.

**Multi-tab: fanout to all connected clients**

`session.ws: WS | null` is replaced with `session.clients: Set<WS>`. Every new WebSocket connection is added to the set; on close it is removed. PTY output is broadcast to all open clients. Input from any tab is forwarded to the PTY — all tabs share the same shell, same as a tmux multi-attach.

Resize follows a last-writer-wins policy: the PTY adopts the dimensions of whichever tab most recently sent a resize message. This matches the behaviour of `tmux attach` from a terminal of different size — acceptable for a local dev tool.

## Considered Options

**Option A: Client-side scrollback buffer (chosen variant: server-side)**

Store the scrollback in `localStorage` or a client-side array. Rejected — the server already owns PTY state; keeping the buffer server-side is consistent with that. It also means the replay works from any browser, not just the one that last had the tab open.

**Option B: PTY output replay via kernel (no buffer)**

Send `SIGWINCH` only, relying on the app to repaint. Works for full-screen TUI apps (vim, Claude Code) but does nothing for shells (bash, fish) where the scrollback is screen output, not app state. Implemented as a complement to the buffer, not a replacement.

**Option C: Persistent scrollback via tmux/screen**

Delegate scrollback to `tmux` or `screen`. The webtty spec explicitly defers long-lived persistence to these tools. Out of scope for this slice.

**Option D: Replay full PTY output from process start**

Keep an unbounded log. Rejected — memory grows without bound for long-lived sessions. A 256 KB ring buffer covers the visible history for typical usage.

**Option E: One active tab at a time (evict on new connect)**

The original behaviour — close the existing WebSocket when a new one connects. Simple, but causes a visible disconnect flash when copying the URL to a new tab. Rejected in favour of the fanout model.

**Option F: Read-only secondary tabs**

Only the first/oldest connected tab can send input; additional tabs are viewers. Rejected — adds complexity for little benefit. Any tab should be fully interactive, same as tmux multi-attach.

## Consequences

- Tab reload restores scrollback for shells (bash, fish) and repaints correctly for TUI apps (vim, htop, Claude Code)
- Welcome banner appears once per session lifetime, not on every reconnect
- Typing `exit` closes all open tabs for that session — no zombie sessions accumulate
- Copying the session URL to a new tab works without disturbing the original tab
- Multiple tabs are fully interactive — input from any tab reaches the PTY
- Resize follows last-writer-wins — the PTY adopts the size of the most recently resized tab
- Memory cost: up to 256 KB per live session (configurable via `SCROLLBACK_MAX` constant in `src/server/session.ts`)
- Scrollback is lost on server restart — acceptable, consistent with session lifetime

## Caveats

**What the scrollback buffer restores**: raw terminal byte output — characters, colors, cursor movement, alternate-screen sequences, and visual selection highlights (all PTY output). This is sufficient for shells (bash, fish) and TUI apps (vim, htop, Claude Code) including their current visual state.

**What it does not restore**:

- *Output older than the ring buffer* — if a session has produced more than 256 KB of output, only the most recent 256 KB is replayed. Content scrolled past that window is gone on reload, same as closing and reopening a tmux window.
- *Cursor blink phase* — ephemeral renderer state, not in PTY output. Harmless.

**Multi-tab sync caveats**:

- Output is broadcast in real time — all tabs see the same PTY output instantly, including visual selection highlights.
- The PTY is shared; the terminal emulator instance is per-tab. If Tab A resizes its window, the PTY adopts Tab A's dimensions (last-writer-wins), which may reflow Tab B's display until Tab B sends its own resize.
- Input from any tab reaches the PTY — in a shared vim session, a keypress from Tab B affects what Tab A sees on the next PTY output cycle.
