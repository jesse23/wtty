# ADR 005: UI — Session Support

**SPEC:** [ui](../specs/ui.md)
**Status:** Accepted
**Date:** 2026-03-22

---

## Context

The bootstrap slice (`src/server.ts`) serves a single hardcoded terminal at `/`. There is no concept of sessions in the UI — every browser tab connects to the same PTY, and there is no way to address a specific session by name.

Two behaviours need to be defined:

1. **Named session URL** — how does a user navigate to a specific session?
2. **Root URL behaviour** — what does `<url>/` do when there may be zero, one, or many sessions?

## Decision

The server handles all session-resolution for browser visits. The SPA is kept simple — it only ever renders a terminal for the session ID in its URL, with no routing logic of its own.

**Server routing**:
- `GET /s/:id` — serves the SPA shell. The session is created by the SPA's WebSocket connection (`/ws/:id`), not on the HTTP GET itself.
- `GET /` — HTTP 302 redirect to `/s/<last-used-id>`. If no sessions exist, creates one named `main` and redirects to `/s/main`. "Last-used" is updated server-side on each WebSocket connect.

**SPA behaviour** (simple, no routing):
- Reads the session ID from `window.location.pathname` (`/s/:id`)
- Opens `WebSocket /ws/:id` — the server creates the PTY session on first connect, or reconnects if it already exists
- No `localStorage`, no API calls on load, no client-side navigation

**Why `main` as the default name?** Short, familiar (git default branch), conveys "primary session". Users who want a different name can rename via `webtty rename`.

**Why track last-used server-side?** The SPA has no state of its own — keeping last-used in the server is consistent with that. It also means the correct session is restored even if the user opens a fresh browser or clears storage.

## Considered Options

### Option A: Server redirects `GET /`; SPA is stateless (chosen)

- **Pros**: SPA stays dead simple — no routing logic, no `localStorage`, no API calls on load. Last-used survives across browsers and cleared storage. Server already owns session state; tracking last-used there is consistent.
- **Cons**: Server holds one extra piece of mutable state (last-used session ID). Lost on server restart — acceptable for a local dev tool.

### Option B: SPA handles all routing via `localStorage`

- **Pros**: Server stays completely stateless for UI concerns.
- **Cons**: SPA needs routing logic, `localStorage` reads, and API calls on load. Last-used is lost if user opens a fresh browser or clears storage. More complexity in the browser for marginal benefit.

### Option C: Root URL always creates a new session

- **Pros**: Stateless — no last-used tracking needed.
- **Cons**: Every page refresh spawns a new PTY. Orphaned sessions accumulate. Confusing UX.

### Option D: Query param (`/?session=main`)

- **Pros**: Single HTML route, no path routing needed.
- **Cons**: Not bookmarkable in the same way, harder to type, inconsistent with REST resource conventions (`/s/:id`).

## Consequences

- SPA is simple: read session ID from URL, open WebSocket, render terminal — nothing else
- `GET /` always resolves to a usable session — new users land at `/s/main` automatically
- Last-used session survives page refresh and fresh browser tabs (server-side state)
- Last-used is reset on server restart — acceptable for a local dev tool
- `webtty run [id]` opens `<url>/s/:id` — consistent with this routing scheme
- Renaming a session invalidates its URL; the server can serve a "session not found" page for the old URL
