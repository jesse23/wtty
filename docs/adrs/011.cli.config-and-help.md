# ADR 011: CLI — config, help, and UX polish

**SPEC:** [cli](../specs/cli.md)
**Status:** Accepted
**Date:** 2026-03-24

---

## Context

Three CLI entry points are missing that users will naturally reach for:

1. **`webtty`** (no arguments) — the most obvious invocation. Currently falls through to Commander's default "display help" behaviour, which is surprising: a user running `npx webtty` for the first time expects something to happen, not a help page.

2. **`webtty help`** — the banner in every terminal session tells users to run `` `bunx webtty help` `` for more information. That command currently fails with "unknown command". The banner promises something that doesn't exist.

3. **`webtty config`** — users need a way to edit the config file without knowing its path. The standard pattern is `$EDITOR`-based.

## Decision

### `webtty` (no args)

1. Start the server if not already running
2. Delegate to `webtty run main`

This is the "just works" entry point — the one you'd put in a README quickstart:

```sh
npx webtty
```

**Why `main` and not a generated ID?** `main` is already the default session created by `GET /`. It is a stable, predictable name. Users who open a second terminal can run `webtty run` with an explicit ID without interfering with `main`.

**`NODE_ENV=test` guard:** same as `openBrowser` — suppressed in test environments to avoid spawning real browser tabs during `bun test`.

### `webtty help`

An explicit alias for `--help` — prints the same Commander-generated help output:

```
Usage: webtty [options] [command]

Web TTY — run terminal sessions in a browser tab
...
```

This makes the banner's call-to-action (`Run \`bunx webtty help\` for more information.`) functional. It does not replace `--help`; both work.

### `webtty config`

Opens `~/.config/webtty/config.json` in the user's preferred editor:

1. Resolve editor: `$VISUAL` → `$EDITOR` → `vi`
2. Spawn the editor with the config path, inheriting stdio
3. Exit when the editor exits

**Why `$VISUAL` fallback?** Some environments set `$VISUAL` for full-screen editors and `$EDITOR` for line editors. Checking both follows the POSIX convention.

**Why `vi` as last resort?** Available on every Unix system. Not `nano` or `vim` — `vi` is the guaranteed-present fallback.

## Considered Options

**Option A: `webtty` opens a new session with a generated ID (not `main`)**

- **Pros**: Each invocation is a fresh session, no risk of reusing a stale session.
- **Cons**: Multiple invocations accumulate zombie sessions. `main` is predictable and idiomatic — consistent with the server's own default redirect behaviour (`GET /` → `main`).

**Option B: `webtty` only starts the server, no browser open**

- **Pros**: Safer, no side effects beyond the server process.
- **Cons**: Requires a second command to actually open a terminal. Defeats the purpose of a frictionless entry point.

**Option C: `webtty help` renders a rich custom help page**

- **Pros**: More control over formatting.
- **Cons**: Maintenance burden — Commander already generates good help output and keeps it in sync with registered commands automatically.

## Consequences

- `npx webtty` / `bunx webtty` becomes the canonical quickstart — one command to go from zero to a browser terminal.
- The banner's `Run \`bunx webtty help\`` call-to-action becomes functional.
- `webtty at main` remains the explicit equivalent — no behaviour change for existing users.
- The no-arg path reuses all existing `at` logic — minimal new code.
- `webtty config` gives users a discoverable way to edit their config without knowing the file path.

## Amendment: help formatting and command surface (2026-03-24)

### Context

Several small UX issues surfaced during review:

- The `Options:` section (`-h, --help`) added noise — `webtty help` is the documented entry point, not `--help`.
- Command descriptions used inconsistent bracket styles (`[id]` vs `<id>`).
- Command order was roughly registration order, not frequency-of-use order.
- The description line (`Web TTY — run terminal sessions in a browser tab`) diverged from the banner title and said the same thing twice with different words.
- `webtty restart` was rarely useful and its destructive side-effect (all sessions lost) made it risky.
- `webtty ls` had no way to narrow output when many sessions exist.
- `webtty run` implied executing a command; the actual behaviour is navigating to a session.
- `isServerRunning()` trusted any HTTP response — a process squatting on the same port would be mistaken for webtty.
- Removing the last session left an idle server running.
- No way to configure clipboard copy behaviour.

### Decisions

**Hide `--help`, keep `webtty help` as canonical.** `program.helpOption(false)` removes the `Options:` section and `[options]` from the usage line. `-h`/`--help` still parses silently for users who reach for it instinctively, but it is not advertised.

**Unify brackets to `[optional]` style.** `<id>` → `[id]` everywhere. Commander's `<required>` vs `[optional]` distinction is correct semantically, but `rm` and `rename` without arguments are handled with explicit runtime guards — so marking them optional in the signature and guarding in the action is equivalent and visually consistent.

**Align parameter columns.** A custom `subcommandTerm` override in `configureHelp` pads short command names (`at`, `rm`, `ls`) to match `mv` width, so `[id]` lands in the same column across all commands with arguments.

**All-caps headings, no colon.** `styleTitle` override uppercases `USAGE` and `COMMANDS` and strips the trailing colon — matches `gh` CLI convention.

**Description first, usage second.** A custom `formatHelp` override renders the description first, then usage with both forms annotated (`webtty` → "Attach to main session and open it", `webtty [command]` → "Execute a specific command"), then commands.

**Command order by frequency of use.** `at`, `ls`, `rm`, `mv` (daily use) → `stop`, `start` (server lifecycle) → `config`, `help` (utility, tail-recognisable).

**Remove `webtty restart`.** `webtty stop && webtty start` is two commands but the intent is transparent and the destructive side-effect (all sessions lost) is explicit. A convenience alias that silently destroys sessions is not worth the surface area.

**`webtty ls [id]` — substring filter.** Accepts an optional argument and filters client-side (`s.id.includes(filter)`). No server changes — the GET /api/sessions endpoint already returns the full list. A glob or regex would be over-engineered for a session list that rarely exceeds a handful of entries.

**`webtty at [id]` (renamed from `run`).** `at` is short for `attach` — the convention used by tmux, zellij, docker, and kubectl for "connect to a session". Aliases `a` and `attach` are registered as undocumented alternatives. Two characters keeps visual alignment with `rm` and `ls`. Researched across 10+ CLIs — `attach` is universal, `a` is the zellij short form; `at` is the best compromise (2 chars, clear etymology, no single-letter ambiguity).

**`webtty mv [id] [new-id]` (renamed from `rename`).** `mv` is the Unix primitive. Aliases `move` and `rename` registered as undocumented alternatives.

**`webtty stop` / `webtty start` (renamed from `down`/`up`).** After researching docker-compose, kubernetes, and session manager CLIs, `stop`/`start` are more universally understood than `up`/`down` for a server lifecycle. `stop`/`start` are kept as primary; `down`/`up` removed.

**`isServerRunning()` validates JSON array.** Previously any HTTP 2xx from the port was accepted, causing false positives when another process (e.g. `ttyd`) occupied the port. Now validates that `GET /api/sessions` returns a JSON array — rejects HTML, 404s, and non-webtty responses.

**`webtty rm` stops server on last session.** The `DELETE /api/sessions/:id` response now includes an `X-Sessions-Remaining` header. The CLI reads it off the same response (zero extra round-trips) and calls `stopServer()` when the count reaches 0.

**`copyOnSelect` and `rightClickBehavior` config keys.** Two independent settings matching VS Code / kitty / Windows Terminal conventions:
- `copyOnSelect: boolean` (default `true`) — auto-copy to clipboard on mouseup, like kitty and Windows Terminal.
- `rightClickBehavior: "default" | "copyPaste"` (default `"default"`) — `"copyPaste"` copies + clears selection on right-click (Windows Terminal style); `"default"` leaves the native context menu untouched. Invalid values silently fall back to `"default"`.
