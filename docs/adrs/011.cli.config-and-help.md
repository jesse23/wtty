# ADR 011: CLI ‚Äî config and help commands

**SPEC:** [cli](../specs/cli.md)
**Status:** Accepted
**Date:** 2026-03-24

---

## Context

Three CLI entry points are missing that users will naturally reach for:

1. **`webtty`** (no arguments) ‚Äî the most obvious invocation. Currently falls through to Commander's default "display help" behaviour, which is surprising: a user running `npx webtty` for the first time expects something to happen, not a help page.

2. **`webtty help`** ‚Äî the banner in every terminal session tells users to run `` `bunx webtty help` `` for more information. That command currently fails with "unknown command". The banner promises something that doesn't exist.

3. **`webtty config`** ‚Äî users need a way to edit the config file without knowing its path. The standard pattern is `$EDITOR`-based.

## Decision

### `webtty` (no args)

1. Start the server if not already running
2. Delegate to `webtty run main`

This is the "just works" entry point ‚Äî the one you'd put in a README quickstart:

```sh
npx webtty
```

**Why `main` and not a generated ID?** `main` is already the default session created by `GET /`. It is a stable, predictable name. Users who open a second terminal can run `webtty run` with an explicit ID without interfering with `main`.

**`NODE_ENV=test` guard:** same as `openBrowser` ‚Äî suppressed in test environments to avoid spawning real browser tabs during `bun test`.

### `webtty help`

An explicit alias for `--help` ‚Äî prints the same Commander-generated help output:

```
Usage: webtty [options] [command]

Web TTY ‚Äî run terminal sessions in a browser tab
...
```

This makes the banner's call-to-action (`Run \`bunx webtty help\` for more information.`) functional. It does not replace `--help`; both work.

### `webtty config`

Opens `~/.config/webtty/config.json` in the user's preferred editor:

1. Resolve editor: `$EDITOR` ‚Üí `$VISUAL` ‚Üí `vi`
2. Spawn the editor with the config path, inheriting stdio
3. Exit when the editor exits

**Why `$VISUAL` fallback?** Some environments set `$VISUAL` for full-screen editors and `$EDITOR` for line editors. Checking both follows the POSIX convention.

**Why `vi` as last resort?** Available on every Unix system. Not `nano` or `vim` ‚Äî `vi` is the guaranteed-present fallback.

## Considered Options

**Option A: `webtty` opens a new session with a generated ID (not `main`)**

- **Pros**: Each invocation is a fresh session, no risk of reusing a stale session.
- **Cons**: Multiple invocations accumulate zombie sessions. `main` is predictable and idiomatic ‚Äî consistent with the server's own default redirect behaviour (`GET /` ‚Üí `main`).

**Option B: `webtty` only starts the server, no browser open**

- **Pros**: Safer, no side effects beyond the server process.
- **Cons**: Requires a second command to actually open a terminal. Defeats the purpose of a frictionless entry point.

**Option C: `webtty help` renders a rich custom help page**

- **Pros**: More control over formatting.
- **Cons**: Maintenance burden ‚Äî Commander already generates good help output and keeps it in sync with registered commands automatically.

## Consequences

- `npx webtty` / `bunx webtty` becomes the canonical quickstart ‚Äî one command to go from zero to a browser terminal.
- The banner's `Run \`bunx webtty help\`` call-to-action becomes functional.
- `webtty run main` remains the explicit equivalent ‚Äî no behaviour change for existing users.
- The no-arg path reuses all existing `run` logic ‚Äî minimal new code.
- `webtty config` gives users a discoverable way to edit their config without knowing the file path.
