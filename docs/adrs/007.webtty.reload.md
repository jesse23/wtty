# ADR 007: webtty — Tab Reload

**SPEC:** [webtty](../specs/webtty.md)
**Status:** Accepted
**Date:** 2026-03-22

---

## Context

When a browser tab is reloaded, the ghostty-web `Terminal` instance is destroyed and recreated from scratch — its scrollback buffer is gone. The PTY on the server remains alive (sessions survive WebSocket disconnects per ADR 005), so the reconnect succeeds, but the user sees a blank terminal with no history.

Three behaviours need to be defined:

1. **Scrollback replay** — how does the fresh terminal recover prior output?
2. **First-connect vs reconnect** — should the welcome banner appear on every connect?
3. **Shell exit** — should the session persist after the user types `exit`?

## Decision

**Scrollback replay (server-side ring buffer)**

The server maintains a `scrollback: string` field per session — a ring buffer of raw PTY output capped at `SCROLLBACK_MAX` bytes (256 KB, ~3000 lines of 80-col output). Every byte from `pty.onData` is appended and the buffer is trimmed with `.slice(-SCROLLBACK_MAX)`.

On reconnect (PTY already exists), the server sends the full scrollback buffer to the fresh WebSocket before any new output arrives. Because the buffer contains raw terminal escape sequences (including alternate-screen sequences like `\x1b[?1049h`), the terminal reconstructs its state correctly — full-screen TUI apps (vim, htop, Claude Code) repaint as expected.

**First-connect vs reconnect**

The welcome banner is shown only on first connect (when `session.pty` is `null`). The banner is also written into `session.scrollback` directly so it appears on replay. On reconnect, no banner is sent — the scrollback replay alone is sufficient.

A `resize` (SIGWINCH) is sent to the PTY immediately after the scrollback replay on every reconnect. This causes TUI apps to redraw their full-screen state on top of the replayed content, ensuring a clean screen for apps like vim that use alternate screen mode. The client also sends an `onopen` resize JSON message, so SIGWINCH fires twice in quick succession — harmless.

**Shell exit closes the session and the tab**

When the PTY exits (e.g. user types `exit`), the server:
1. Deletes the session from the registry immediately
2. Closes the WebSocket with code `4001`

The client's existing `4001` handler (introduced in ADR 005 for deleted sessions) writes "Session removed." and calls `window.close()` after 2s. No delay is added before closing — the user explicitly typed `exit`, so immediate closure is the right behaviour.

## Considered Options

**Option A: Client-side scrollback buffer (chosen variant: server-side)**

Store the scrollback in `localStorage` or a client-side array. Rejected — the server already owns PTY state; keeping the buffer server-side is consistent with that. It also means the replay works from any browser, not just the one that last had the tab open.

**Option B: PTY output replay via kernel (no buffer)**

Send `SIGWINCH` only, relying on the app to repaint. Works for full-screen TUI apps (vim, Claude Code) but does nothing for shells (bash, fish) where the scrollback is screen output, not app state. Implemented as a complement to the buffer, not a replacement.

**Option C: Persistent scrollback via tmux/screen**

Delegate scrollback to `tmux` or `screen`. The webtty spec explicitly defers long-lived persistence to these tools. Out of scope for this slice.

**Option D: Replay full PTY output from process start**

Keep an unbounded log. Rejected — memory grows without bound for long-lived sessions. A 256 KB ring buffer covers the visible history for typical usage.

## Consequences

- Tab reload restores scrollback for shells (bash, fish) and repaints correctly for TUI apps (vim, htop, Claude Code)
- Welcome banner appears once per session lifetime, not on every reconnect
- Typing `exit` closes the browser tab — no zombie sessions accumulate
- Memory cost: up to 256 KB per live session (configurable via `SCROLLBACK_MAX` constant in `server.ts`)
- Scrollback is lost on server restart — acceptable, consistent with session lifetime
